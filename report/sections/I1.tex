\subsection{I.1 Reproducing the Baseline}

\paragraph{Run identity.}
Commit \texttt{7e696c42} on branch \texttt{baseline-fls}, W\&B run
\href{https://wandb.ai/felsomoye-university-of-cambridge/tunix/runs/jgs4c6kl}{\texttt{jgs4c6kl}},
wall-clock 4h\,41m, completing 3364 GRPO steps (full schedule $3738\times0.9$).

\paragraph{Baseline patches.}
The baseline did not run as shipped because TFDS/GSM8K loading failed with a protobuf \texttt{FieldDescriptor.label} error, caused by a \texttt{protobuf} 7.x and \texttt{tensorflow-datasets} 4.9.9 compatibility issue. The fix was to pin \texttt{protobuf==6.31.1} in \texttt{requirements.txt}; this changed only dependency plumbing, not the model, reward functions, data split, or training hyperparameters.

\paragraph{Accuracy.}
On the GSM8K test split (seed-42 shuffle, first 64 examples, greedy decoding) the base
\texttt{gemma-3-1b-it} scores $33/64=51.56\%$ exact, while the baseline LoRA GRPO checkpoint
(step 3364) scores only $2/64=3.12\%$ --- substantially worse than base, indicating a training
failure rather than a checkpoint-restore bug.

Eval reward peaks around 6.8 near step 460, then declines through an oscillating descent to
about $-1$ by step $\sim$2750 and flatlines; KL holds near 0.5 with spikes around steps 2500
and 3300. The policy learned reward-correlated behaviour early but drifted from the reference
without preserving GSM8K accuracy.
